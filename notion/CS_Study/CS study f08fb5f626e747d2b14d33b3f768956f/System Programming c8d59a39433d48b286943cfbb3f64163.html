Memory Structure

code

also called “text section”

code of executing program

data

store global, static variables

constructed and destructed while program starts and ends

heap

managed by user

allocated and freed dynamically

located from low to high address

  • pros
    • global access
    • no space limit
  • cons
    • slow access
    • memory fragment (need to be managed by user)

stack

local variables and parameters

LIFO

located from high to low address

  • pros
    • fast access
    • auto free, managed by CPU
  • cons
    • space limit
    • size is not changable

Process vs Thread

Process

CPU 자원 할당의 단위

sections : text, data, bss, etc…

  • text
    • executable code, read-only data
  • data
    • initialized data (read-write)
  • bss
    • uninitialized data (mapped

resources : timers, pending signals, open files, network connections…

stored inside the kernel in the process’s process descriptor

Thread

프로세스 내 수행 단위 (unit of activity : Linux System Programming p17)

stack (stores local variable), processor state, current location in the object code

thread ID, pc(program counter)

Process hierarchy

identified as PID (process ID)

fork() : duplicate the calling process (parent), create new process(child)

Child is reparented to the init process by kernel if the parent is killed earlier

When child process terminated, parts of its resident in memory are kept - allow the parent to inquire(wait()) about its status upon terminating

zombie process : terminated but not waited

Synchronization

Sync vs Async

sync : simultaneously, one blocked until others returned

async : assign task to event queue or background thread, and start the code following, not returned

Critical Section

the section for accessing same resources

requirement

mutual exclusion : only single process executed on

progress : processes with no action only can be in

bounded waiting : the number of entry of other processes while applying one process is limited

Solutions :

  • mutex lock
    • limitation : time efficiency on multi-process
  • semaphores
    • countable permissions to give processes
    • binary semaphores ~= mutex
    • spinlock : looping while available lock checked → efficient for short term blocking
  • deadlock
    • case that multiple threads are waiting for unlock
